FinOps · Database · Runtime Split

DB·인프라 비용 구조 재설계

상황

r5.4xlarge급 DB에 SP, SQL Agent, 반복 배치성 작업이 함께 몰리면서 CPU 사용률과 비용이 동시에 커지고 있었습니다.

접근

SP와 Agent 작업을 먼저 분석해 튜닝했고, DB에서 처리하지 않아도 되는 일부 작업은 애플리케이션 프로세스로 분리했습니다. 데이터베이스는 데이터 정합성과 조회에 집중하도록 역할을 좁혔습니다.

결과

전체 CPU 사용률을 50% 이상 줄이고 더 작은 인스턴스 운영이 가능해졌습니다. DB뿐 아니라 애플리케이션, 배치, 모니터링, 스토리지까지 함께 조정해 인프라 비용을 약 40% 절감했습니다.

Legacy · Security · Streaming

VB 6.0 시스템의 마이크로 3-Tier 전환

상황

오래된 VB 6.0 클라이언트가 DB에 직접 연결하는 구조라 보안 리스크와 DB 부하가 함께 존재했습니다. 전체 클라이언트를 한 번에 갈아엎기는 어려웠습니다.

접근

COM DLL 기반 게이트웨이 프록시를 두어 기존 클라이언트 수정 범위를 줄였고, 대용량 Record Set 응답은 NDJSON HTTP 스트리밍과 생산자-소비자 구조로 처리했습니다.

결과

DB 직접 연결을 차단하면서도 기존 업무 흐름은 유지했습니다. 보안 경계를 분리하고 대용량 응답 처리의 안정성을 높였습니다.

Operating Method

제가 복잡한 시스템을 다루는 방식

기술을 먼저 고르기보다, 비용이 새는 지점과 장애가 번지는 경계를 먼저 찾습니다.

운영에서 버티는 코드, 추적 가능한 장애, 작게 나눈 책임, 검증 가능한 변경을 중요하게 생각합니다. 새 시스템을 만들 때도 로그, 권한, 배포, 유지보수 흐름을 처음부터 함께 설계합니다.

01

계측 가능한 문제로 바꿉니다

CPU, 요청 수, 응답 지연, 수동 처리 시간, 장애 전파 범위를 먼저 확인해 개선 대상을 숫자로 고정합니다.

02

경계를 다시 나눕니다

DB가 해야 할 일, 애플리케이션이 가져가야 할 일, 배치로 분리할 일을 나눠 병목이 한 곳에 쌓이지 않게 합니다.

03

기존 업무를 멈추지 않습니다

레거시를 한 번에 버리기보다 게이트웨이, 프록시, 스트리밍, 동기화 계층으로 안전하게 전환합니다.

04

운영 자산으로 남깁니다

대시보드, 로그, 코드 리뷰, 문서화, OKR 흐름까지 연결해 개선이 한 번의 프로젝트로 끝나지 않게 합니다.

05

AI를 검토 파트너로 씁니다

코드베이스 탐색, 변경 영향 분석, 문서 초안, 테스트 관점 정리에 AI를 붙여 반복 시간을 줄이고 판단은 로그와 코드로 검증합니다.

06

작게 나누고 계속 검증합니다

큰 전환도 기능·책임 단위로 쪼개고, 각 단계마다 빌드와 테스트로 확인해 운영 가능한 상태를 유지합니다.
